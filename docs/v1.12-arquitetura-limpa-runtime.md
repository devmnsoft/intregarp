# v1.12 — arquitetura limpa do runtime operacional

## Mudança entregue

O antigo `V110RuntimeController` deixou de concentrar SQL e acesso direto a PostgreSQL. Ele permanece apenas como compatibilidade obsoleta em `/api/v110-runtime`, protegido por autenticação.

## Fluxo adotado

1. Controllers específicos recebem a requisição HTTP e extraem o tenant via `IntegraControllerBase`.
2. `OperationalRuntimeUseCases` valida tenant, retorna `Result<T>` e orquestra o caso de uso.
3. `IOperationalRuntimeRepository` define o contrato de persistência.
4. `PostgresV112OperationalRepository` concentra o SQL parametrizado com `tenant_id` obrigatório na Infrastructure.

## Controllers criados

- `CustomersController`
- `ProductsController`
- `InventoryController`
- `OrdersController`
- `TasksController`

Os controllers operacionais usam `[Authorize]`, não importam Npgsql/Dapper e não contêm SQL inline.

## Limitações conhecidas

A consolidação completa de todos os CRUDs dedicados, delete por soft delete, buscas por id e todos os fluxos de billing/outbox/templates deve ser validada em ambiente com SDK .NET e PostgreSQL disponíveis. Este container não possui SDK .NET nem PowerShell para executar as validações finais locais.
