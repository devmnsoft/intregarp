# IntegraRP v1.5 — CRUDs operacionais, jornada completa, QA e deploy assistido

## Objetivo
A v1.5 torna a solução testável em ambiente local, Docker e Windows conectando banco PostgreSQL, API, Web, Worker, Mobile e jornada do cliente sem criar um novo sistema e sem trocar o schema `integrarp`.

## Escopo operacional
- Core/Admin: tenants, filiais, setores, cargos, usuários, perfis, permissões e auditoria.
- Comercial: clientes, contatos, endereços, leads e oportunidades simples.
- Estoque: produtos, categorias, fornecedores, locais, entradas, movimentos, reservas e estoque crítico.
- Pedidos: criação, edição, itens, validação de estoque, confirmação, cancelamento e histórico.
- Flow/Tarefas: processos, publicação, instâncias, fila, assumir, comentar, concluir e timeline.
- Faturamento: fatura a partir do pedido, título, boleto fake, vencidos e consultas.
- Connect/Outbox: templates, mensagem fake, processamento, reprocessamento e histórico.
- Project/Jornada: board, kanban, cards, onboarding, progresso, ações recomendadas e auditoria.

## Banco de dados
Execute `database/scriptcompleto.sql`. O script cria `integrarp.v15_operational_object`, `integrarp.v15_customer_full_journey_check`, `integrarp.v15_worker_queue_health`, views `integrarp.vw_v15_customer_full_journey` e `integrarp.vw_v15_operational_readiness`, além da migration `0017_v15_validacao_real_cruds_qa_deploy`.

## API e validação
Endpoints de validação:
- `GET /api/validation/flow/customer-full-journey`
- `GET /api/validation/flow/order-to-billing-demo`
- `GET /api/validation/scriptcompleto/status`

## Web
As telas devem consumir APIs reais, exibir loading, tratar erros, 401, 403, vazio inteligente, ajuda contextual e próxima ação recomendada. Rotas principais: `/admin`, `/customers`, `/products`, `/inventory`, `/orders`, `/flow`, `/billing`, `/connect`, `/project`, `/getting-started` e `/journey/what-to-do-now`.

## Worker
O Worker processa outbox, reprocessa erros, gera ações recomendadas, atualiza score de adoção, detecta tarefas atrasadas, recalcula KPIs e registra falhas sem derrubar o serviço.

## Mobile
O Mobile consome API real para login, minha fila, detalhe e conclusão de tarefa, o que fazer agora e sincronização simples. Validar com `npm run typecheck` em `apps/IntegraRP.Mobile`.

## Demo v1.5
1. Subir PostgreSQL.
2. Executar `database/scriptcompleto.sql`.
3. Subir API, Web e Worker.
4. Fazer login.
5. Criar setor, usuário, cliente e produto.
6. Registrar entrada de estoque.
7. Criar pedido, adicionar item e confirmar.
8. Ver e concluir tarefa.
9. Gerar fatura, título e boleto fake.
10. Processar outbox.
11. Ver dashboard, próxima ação, IA e auditoria.
12. Rodar `GET /api/validation/flow/customer-full-journey`.

## QA
Casos manuais e automatizados cobrem login, tenant, usuários, setores, clientes, produtos, estoque, pedido, Flow, tarefa, faturamento, outbox, dashboard, jornada, IA, auditoria, Worker e Mobile. Também validar isolamento Tenant A/B, RBAC 401/403, ausência de SQL em controllers e ausência de Dapper fora de Infrastructure.

## Windows
Usar scripts em `scripts/*-windows.ps1` ou `scripts/*-windows.cmd`: configurar ambiente, migrar banco, executar API, Web, Worker e testes.

## Docker
Quando Docker estiver disponível: `docker compose build`, `docker compose up -d`, `docker compose ps`; depois aplicar `database/scriptcompleto.sql` e validar Swagger/API.

## Limitações conhecidas
Neste ambiente de execução o SDK `dotnet` e o Docker não estavam disponíveis. O typecheck mobile foi executado com sucesso.
