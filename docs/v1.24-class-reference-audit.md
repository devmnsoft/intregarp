# IntegraRP v1.24 — auditoria de classes e referências

Auditoria executada por inspeção estática com `find`, `rg`, leitura dos arquivos principais de Web/API/Auth e diagnóstico de build bloqueado pela ausência de SDK .NET.

## Achados corrigidos neste PR

| Categoria | Arquivo | Achado | Correção |
| --- | --- | --- | --- |
| Controller sem integração correta | `src/IntegraRP.Web/Controllers/AccountController.cs` | Login chamava `/auth/login`, montava payload com `tenant` e criava role fixa. | Refatorado para `IIdentityApiClient`, `/api/auth/login`, payload tipado e claims reais. |
| Interface sem implementação | Web Identity | Não existiam `IIdentityApiClient`, `IIdentitySessionStore` e serviço de usuário corrente. | Criadas implementações tipadas e registradas no DI. |
| Singleton/scoped | `ProtectedIdentitySessionStore` | Necessidade de token store server-side sem dependência scoped. | Implementado como singleton com Data Protection e dicionário concorrente protegido. |
| Pipeline incorreto | `src/IntegraRP.Api/Program.cs` | Middleware global vinha após autenticação/autorização/logging. | Reordenado para iniciar com tratamento global, correlation id e logging. |
| Health checks superficiais | API | Rotas mapeavam health genérico sem validar dependências. | Adicionados checks de PostgreSQL, schema/migration 0030 e storage local. |
| Schema duplicado | `database/migrations/0029_v123_core_operacional_real.sql` | Lockout paralelo (`failed_login_count`, `lockout_until`, `auth_login_attempt`). | Migration 0030 migra para contrato canônico sem apagar dados. |

## Achados pendentes registrados honestamente

- Ainda existem JavaScripts legados com `innerHTML` e diálogos nativos (`alert`, `confirm`, `prompt`) em módulos fora do núcleo de autenticação v1.24.
- Não foi possível validar registros DI por compilação porque `dotnet` não está instalado no ambiente.
- Não foi possível executar testes de arquitetura .NET pelo mesmo bloqueio.
- Há controllers e telas históricas de módulos avançados que permanecem no repositório, mas a v1.24 não expandiu esses módulos.

## Regras de regressão adicionadas por implementação

- Fallback policy Web exige usuário autenticado por padrão.
- Login/forgot/reset/access denied/session expired são explicitamente anônimos.
- Logout GET foi removido; apenas POST com antiforgery permanece.
- API readiness exige migration `0030` em `integrarp.schema_migrations`.
